# ระบบสรุป: หรีดร่วมบุญ (rrb.center)
> เอกสารฉบับนี้อธิบายภาพรวมระบบ เพื่อใช้ส่งต่อ วางระบบต่อ และทำ Storyboard ขั้นตอนทำงาน
> อัปเดตล่าสุด: 17 มิถุนายน 2569

---

## 1. ภาพรวมโครงการ

**หรีดร่วมบุญ** คือแพลตฟอร์มออนไลน์สำหรับ **ร่วมบุญเป็นเงินแทนการส่งหรีด** ในงานศพ  
แทนที่จะส่งหรีดดอกไม้ที่สร้างขยะ — ผู้ร่วมงานโอนเงินออนไลน์ได้เลย  
ระบบจะพิมพ์ **ป้ายชื่อ (nameplate)** ออกมาติดบนกระดานแทนหรีดจริง

**Tagline**: ร่วมอาลัย · ร่วมทำบุญ · ร่วมลดขยะ

**URL**: https://rrb.center

---

## 2. ผู้ใช้งาน (User Roles)

| บทบาท | คือใคร | เข้าระบบด้วยอะไร | ทำอะไรได้ |
|---|---|---|---|
| **ผู้ร่วมบุญ (Donor)** | คนทั่วไปที่รู้จักงานศพ | ไม่ต้อง login | อัปโหลดสลิป → กรอกชื่อ → ดูป้ายชื่อ |
| **เจ้าภาพ (Host)** | ครอบครัว/ผู้จัดงาน | รหัสพิเศษ (host_code) | ดูรายชื่อผู้ร่วมบุญ / ยอดเงิน / รายงาน |
| **ศูนย์ (Center Manager)** | เจ้าหน้าที่ศูนย์บริหารหรีด | รหัสศูนย์ (center_code) | เปิด-ปิดงานศพ / ติดตามภาพรวม / โอนเงินให้เจ้าภาพ |
| **ผู้ดูแลระบบ (Super Admin)** | ส่วนกลาง | รหัสผ่าน (ADMIN_PASSWORD) | จัดการทุกศูนย์ / รายงานภาพรวม / สิทธิ์ผู้ใช้ |

## 3. โครงสร้างข้อมูลหลัก

```
centers (ศูนย์)
  └── memorials (งานศพ)
        └── donations (การร่วมบุญ)
              └── nameplates (ป้ายชื่อ)
                    └── print_jobs (คิวพิมพ์)
```

### 3.1 ศูนย์ (centers)
- 1 ศูนย์ = 1 สาขา เช่น "ศูนย์บริหารหรีดร่วมบุญ อบต.บึง"
- มีรหัสศูนย์ (center_code) และรหัส อปท. 8 หลัก (official_lgo_code)
- เบอร์โทรศูนย์ = ใช้ generate QR PromptPay ให้ผู้ร่วมบุญโอนเงิน

### 3.2 งานศพ (memorials)
- 1 งาน = 1 ผู้วายชนม์
- มี URL เฉพาะ เช่น `rrb.center/evt-2026-xxxx`
- สถานะ: `draft` → `active` → `closed`
- เจ้าภาพแต่ละงานมีรหัสส่วนตัว (host_code) เช่น "HABCD"

### 3.3 การร่วมบุญ (donations)
- 1 รายการ = 1 การโอนเงิน
- **ยืนยันอัตโนมัติ** (ไม่ต้องรอตรวจสลิป)
- มีการตรวจจับสลิปซ้ำด้วย SHA256 hash
- สถานะป้ายชื่อ: `pending` → `queued` → `printed` → `posted`

### 4.1 Flow ผู้ร่วมบุญ (Donor) — หน้าสาธารณะ

```
[เข้า URL งาน]
      ↓
/{slug}  →  ดูข้อมูลผู้วายชนม์ / วันฌาปนกิจ / ที่ตั้ง
      ↓
/{slug}/payment  →  สแกน QR โอนเงิน / ดูเลขบัญชี
                     อัปโหลดสลิป (ตรวจหาสลิปซ้ำ)
      ↓
/{slug}/verifying  →  รอการตรวจสอบ (auto redirect เร็วๆ)
      ↓
/{slug}/print-name  →  กรอกชื่อจริง + ตำแหน่ง (กรอกครั้งเดียว ไม่แก้ได้)
                        กด "ร่วมบุญ" → สร้าง donation (auto-confirm ทันที)
      ↓
/{slug}/printing  →  ดูป้ายชื่อ e-nameplate (ส่งคิวพิมพ์จริง)
      ↓
/print-done  →  "ขอบคุณที่ร่วมบุญ"
```

**หมายเหตุสำคัญ**:
- ชื่อผู้ร่วมบุญกรอกได้ครั้งเดียว — ป้องกัน bug ชื่อไม่อัปเดต
- ไม่ต้อง login ตลอด flow
- หลัง submit donation ระบบ auto-confirm ทันที (ไม่รอตรวจสลิป)

### 4.3 Flow ศูนย์บริหาร (Center Manager)

> **หลักการสำคัญ**: ศูนย์ไม่อนุมัติรายการย่อยระหว่างงาน  
> ทุก flow ผู้ร่วมบุญ (upload slip → กรอกชื่อ → พิมพ์ป้าย) ทำงานอัตโนมัติ  
> ศูนย์มีบทบาทควบคุม 3 ด้าน หลังจบงานและในเชิงบริหาร

**บทบาทที่แท้จริงของ Dashboard ศูนย์**

**① อนุมัติการโอนเงินให้เจ้าภาพ (หลังจบงาน)**
```
ศูนย์กดสรุปงาน → ระบบคำนวณ:
  - ยอดร่วมบุญรวมทั้งหมด
  - จำนวนรายการร่วมบุญ
  - ค่าดำเนินการของระบบ (100 บาท/ราย)
  - ยอดสุทธิที่ต้องโอนให้เจ้าภาพ
  - บัญชีปลายทางเจ้าภาพ (ที่กรอกไว้ล่วงหน้า)

→ ศูนย์ตรวจสอบและ "อนุมัติ" การโอนเงินออกจากบัญชีศูนย์
→ ศูนย์รับผิดชอบโอนยอดสุทธิให้เจ้าภาพด้วยตนเอง
```

**② อนุมัติสิทธิ์สมาชิกและแอดมินศูนย์ (บริหารศูนย์)**
```
ศูนย์กำหนดว่าใครทำอะไรได้บ้าง:
  - ใครเป็นแอดมินศูนย์
  - ใครเปิดงานใหม่ได้
  - ใครแก้ไขข้อมูลงานได้
  - ใครดูรายงานได้
  - ใครกดปิดงานได้
  - ใครกดอนุมัติสรุปโอนเงินได้
  - ใครส่งรายงานให้ อปท. ได้
```

**③ อนุมัติข้อมูลสรุปส่งให้ อปท. (รายงานเจ้าของโครงการ)**
```
หลังจบงาน ศูนย์ตรวจสอบและอนุมัติรายงานเพื่อส่งให้ อปท.:
  - จำนวนงานศพที่ใช้ระบบ
  - จำนวนผู้ร่วมบุญ
  - ยอดเงินร่วมบุญรวม
  - ยอดเงินสุทธิที่ส่งถึงเจ้าภาพ
  - จำนวนพวงหรีดที่ลดได้โดยประมาณ
  - รายงานลดขยะ / Zero Waste / ESG / LPA / จังหวัดสะอาด

อปท. = เจ้าของนโยบายและผู้สนับสนุนศูนย์ → ต้องเห็นผลลัพธ์โครงการ
```


## 5. การคำนวณการเงิน

```
ตัวอย่าง: งานมีผู้ร่วมบุญ 10 ราย เฉลี่ยรายละ 500 บาท

  ยอดรวมที่รับได้:       5,000 บาท
  ค่าบริการ (10 × 100): - 1,000 บาท
                         ──────────
  ยอดสุทธิให้เจ้าภาพ:    4,000 บาท
```

- เงินร่วมบุญไหลเข้าบัญชีธนาคารของศูนย์ประจำ อปท
- หลังปิดงาน เจ้าภาพกรอกบัญชีตัวเอง → ศูนย์โอนยอดสุทธิให้

## 6. ระบบพิมพ์ป้ายชื่อ (Nameplate Printing)

```
[Donor กด "ร่วมบุญ"]
      ↓
donation created (DB) + outbox_job created (atomic transaction)
      ↓
Worker ดึง outbox_job → ส่ง print request ไป PRINT_SERVICE_URL
      ↓
เครื่องพิมพ์รับ job → พิมพ์ป้ายชื่อจริง
      ↓
Update: nameplate_status = "printed"
      ↓
เจ้าหน้าที่ติดป้ายบนกระดาน → Update: "posted"
```

**ระบบ Outbox Pattern**: ป้องกัน donation หายหาก printer timeout  
- Max 5 retry attempts
- ถ้า printer ไม่ตอบ → donation ยังบันทึกสำเร็จ ป้ายค้างในคิว
- Worker ดึงงานค้างไป retry อัตโนมัติ

## 7. การตรวจจับสลิปซ้ำ (Duplicate Slip Detection)

```
Donor อัปโหลดสลิป
      ↓
Server คำนวณ SHA256 hash ของไฟล์
      ↓
เทียบกับ slip_submissions ของงานนี้
      ↓
ไม่ซ้ำ → บันทึกปกติ
ซ้ำ → บันทึกแต่ตั้ง flag: slip_duplicate_warning = true
      ↓
แสดงเตือนในแดชบอร์ดเจ้าภาพ (สีเหลือง)
Admin สามารถ reject ได้ถ้าต้องการ
```

## 11. สรุป Business Logic สำคัญ

1. **1 ศูนย์ หลาย งานศพ** — ศูนย์เป็น tenant, งานศพเป็น event
2. **เจ้าภาพ ≠ ศูนย์** — เจ้าภาพเข้าระบบด้วย host_code แยกต่างหาก
3. **Auto-confirm** — donation ทุกรายการยืนยันอัตโนมัติ (ไม่ต้องรอตรวจสลิป)
4. **ชื่อผู้ร่วมบุญกรอกครั้งเดียว** — ป้องกัน bug ชื่อบนป้ายผิด
5. **ค่าบริการ 100 บาท/ราย** — หักออกตอนปิดงาน ไม่หักจากยอดโอน
6. **QR PromptPay** — generate จากเบอร์ศูนย์ หรือใช้รูปภาพ QR ที่อัปโหลดไว้
7. **Nameplate → Board** — ป้ายชื่อพิมพ์จริงและติดบนกระดานแทนหรีด
8. **Slug = URL สาธารณะ** — แชร์ link ให้คนมาร่วมบุญ เช่น `rrb.center/evt-2026-xxxx`

## 12. Storyboard ขั้นตอนทำงาน (ภาพรวม)

```
ขั้นที่ 1: ศูนย์เปิดงาน
  ─────────────────────
  ศูนย์ login → กรอกข้อมูลผู้วายชนม์ → ระบบสร้าง slug + host_code
  ส่ง link (/slug) และ host_code ให้เจ้าภาพ

ขั้นที่ 2: เจ้าภาพรับข้อมูล
  ───────────────────────────
  เจ้าภาพ login ด้วย host_code → กรอกบัญชีธนาคารตัวเอง + เอกสาร
  แชร์ link งานให้ญาติมิตร

ขั้นที่ 3: ผู้ร่วมบุญโอนเงิน
  ──────────────────────────────
  คลิก link → ดูข้อมูลงาน → สแกน QR โอนเงิน
  อัปโหลดสลิป → กรอกชื่อ → ได้ป้ายชื่อ e-nameplate
  ระบบพิมพ์ป้ายชื่อจริงอัตโนมัติ

ขั้นที่ 4: ระหว่างงาน
  ────────────────────
  เจ้าภาพดู tab "รายชื่อ" แบบ real-time
  ศูนย์ติดป้ายชื่อบนกระดาน (update: "posted")
  แอดมินอ่านรายชื่อ "tab รายงาน" ในงาน

ขั้นที่ 5: ปิดงาน
  ─────────────────
  ศูนย์กด "ปิดงาน" → สร้างรายงานสรุป
  คำนวณยอดสุทธิ (หักค่าบริการ 100 บาท/ราย)
  ศูนย์โอนเงินสุทธิให้เจ้าภาพ
```

## 13. ข้อจำกัดและแผนต่อไป

### ปัจจุบัน (MVP)
- Login แบบ password/code ง่าย (ยังไม่มี JWT/OTP)
- ยังไม่มี payment gateway จริง (รับสลิปแล้ว verify เอง)
- ยังไม่ได้เชื่อม printer service จริง (outbox พร้อมแล้ว รอ connect)

### แผนต่อ
- Line/Google OAuth login
- Payment gateway (PromptPay API / Omise)
- เชื่อม printer service จริง
- Mobile app
- ระบบ SMS แจ้งเตือนเจ้าภาพ
